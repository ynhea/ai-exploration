# 3주차: AI 엔지니어 직무 체험 커리큘럼

## 프로젝트 개요

**가상 회사: 컴퍼스랩(CompassLab)** — 시리즈 A 단계의 HR테크 스타트업
**프로젝트: 사내 AI 어시스턴트 "컴퍼스봇" 구축**

당신은 컴퍼스랩에 AI 엔지니어 인턴으로 합류했습니다. 첫 프로젝트는 신입사원 온보딩 부담을 줄이기 위한 사내 AI 챗봇을 MVP부터 시작해 실제 서비스 수준까지 발전시키는 것입니다. Day 1의 단순 챗봇이 Day 7에는 문서 검색 + 에이전트 기능 + 외부 도구 연동까지 갖춘 서비스로 진화합니다.

**공통 원칙**
- 매일 문제 → 직접 구현 → 막히는 지점에서 최소 개념 학습 → 개선 순서로 진행
- 코드는 스켈레톤/힌트 위주로 제공, 정답은 최후 단계에서만 공개
- Must만 완료해도 그날의 직무를 체험한 것으로 인정

---

## Day 1. 온보딩 챗봇 MVP — Backend Engineer 역할

**오늘의 회사 및 서비스**
컴퍼스랩 HR팀이 "신입사원이 사규를 계속 물어봐서 힘들다"며 급하게 챗봇을 요청했습니다. 오늘 안에 동작하는 프로토타입을 보여줘야 합니다.

**오늘의 직무**: AI Backend Engineer (LLM API 연동 담당)

**오늘 해결해야 하는 업무**
HR팀이 웹에서 질문을 입력하면 답변이 돌아오는 최소 기능 챗봇을 오늘 중으로 완성하라는 요청을 받았습니다.

**실습 목표**
OpenAI API를 FastAPI 백엔드에 연동하고, 간단한 웹 UI에서 요청-응답이 오가는 구조를 직접 만든다.

**Must (2~3시간)**
- FastAPI 프로젝트 초기화, `/chat` 엔드포인트 설계
- OpenAI API 키 발급 및 환경변수 관리(.env, .gitignore)
- 사용자 입력 → API 호출 → 응답 반환 흐름 구현
- 간단한 HTML/JS 채팅 UI 연결 (프레임워크 없이 fetch만 사용해도 무방)

**Should**
- 대화 히스토리(멀티턴) 유지 기능 추가 — 세션 단위로 메시지 리스트 관리
- 스트리밍 응답(Server-Sent Events) 적용해서 타이핑 효과 구현

**Could**
- Docker로 컨테이너화해서 `docker run` 한 줄로 실행 가능하게 만들기
- 간단한 로깅(요청/응답/토큰 수) 추가

**트러블슈팅 (의도적으로 겪게 될 문제)**
- API 키 노출로 인한 401 에러, `.env` 로드 실패
- CORS 에러 (프론트-백엔드 분리 시)
- Rate limit / timeout 에러 처리

**왜 이 업무를 하는가**
실무 AI 엔지니어 업무의 상당수는 "이론적으로 완벽한 모델"이 아니라 "빠르게 동작하는 MVP"를 만드는 일입니다. API 연동, 환경 설정, 기본 에러 핸들링은 모든 LLM 서비스의 출발점입니다.

**배우게 되는 핵심 역량**
API 연동 설계, 환경변수/시크릿 관리, 동기/비동기 처리, 기본적인 웹 통신 구조

**오늘 배운 AI 개념**
LLM API 호출 구조(messages, role, system prompt), 토큰(token)이란 무엇인가, temperature 등 기본 파라미터

**오늘의 회고 (적성 판단 질문)**
- API 연동과 서버 구조를 짜는 과정이 흥미로웠는가, 지루했는가?
- 에러를 만났을 때 짜증이 났는가, 오히려 흥미로웠는가?

## Day 2. 프롬프트 품질 개선 — Prompt/AI Engineer 역할

**오늘의 회사 및 서비스**
어제 만든 챗봇을 HR팀이 써봤더니 "질문과 상관없는 엉뚱한 답을 한다", "회사 정책과 다른 답을 지어낸다(할루시네이션)"는 피드백이 들어왔습니다.

**오늘의 직무**: Prompt Engineer

**오늘 해결해야 하는 업무**
동일한 코드베이스에서 프롬프트만 개선하여 답변 품질을 높이라는 요청. 아직 RAG는 없는 상태(내부 문서 연결 전)이므로, 시스템 프롬프트 설계로 얼마나 개선 가능한지 실험합니다.

**실습 목표**
시스템 프롬프트 설계, few-shot 예시, 응답 형식 제약을 통해 답변 품질과 일관성을 높인다.

**Must**
- 나쁜 프롬프트 vs 개선된 프롬프트 비교 실험 (최소 5개 질문 세트로 테스트)
- 역할(persona), 제약조건, 출력 형식을 명시한 시스템 프롬프트 작성
- "모르면 모른다고 답하라"는 지침으로 할루시네이션 유도 상황 재현 및 완화

**Should**
- Few-shot 예시 3~5개를 추가해 답변 톤/형식 일관성 개선
- 프롬프트 버전을 파일로 관리(v1, v2, v3)하고 결과를 비교표로 정리

**Could**
- 간단한 A/B 테스트 스크립트 작성 (같은 질문을 두 프롬프트에 보내 결과 자동 비교)

**트러블슈팅**
- 프롬프트를 아무리 바꿔도 해결 안 되는 문제 직접 겪기 (→ Day 3의 RAG 필요성 자연스럽게 체감)
- 프롬프트가 길어지며 토큰 비용/속도 저하 체감

**왜 이 업무를 하는가**
실무에서 모델을 재학습시키는 경우는 드뭅니다. 대부분의 품질 개선은 프롬프트 설계로 이루어지며, 이 한계를 몸소 느껴야 RAG/파인튜닝 등 다음 단계의 필요성을 이해할 수 있습니다.

**배우게 되는 핵심 역량**
프롬프트 설계, 테스트 케이스 기반 검증, 정성적 품질 평가 방법

**오늘 배운 AI 개념**
시스템 프롬프트, few-shot learning, 할루시네이션(hallucination), 프롬프트의 한계

**오늘의 회고**
- 프롬프트 개선처럼 "미묘한 것을 조율하는" 작업이 나에게 맞는가?
- 정량적 정답이 없는 작업에서 오는 답답함/재미는 어땠는가?

## Day 3. 사내 문서 검색 RAG 구축 — AI Engineer (Search) 역할

**오늘의 회사 및 서비스**
프롬프트만으로는 한계가 명확했습니다. HR팀이 실제 "취업규칙.pdf", "복지제도.md" 같은 사내 문서를 챗봇이 참고해서 답하길 원합니다.

**오늘의 직무**: AI Engineer (RAG 담당)

**오늘 해결해야 하는 업무**
실제 문서(2~3개, 직접 준비하거나 더미 사내 문서 작성)를 챗봇이 검색해서 답변에 활용하도록 만들기.

**실습 목표**
문서를 chunk로 나누고 embedding으로 변환, vector DB에 저장한 뒤 질문과 유사한 chunk를 검색해 LLM에 함께 전달하는 RAG 파이프라인을 처음부터 구현한다.

**Must**
- 더미 사내 문서 2~3개 작성(취업규칙, 복지제도, 휴가정책 등)
- 문서를 chunk 단위로 분할
- OpenAI Embedding API로 chunk를 벡터화
- ChromaDB(로컬 vector DB)에 저장 및 유사도 검색 구현
- 검색된 chunk를 프롬프트에 삽입해 답변 생성 (Day 1~2 챗봇과 연결)

**Should**
- chunk 크기/overlap을 바꿔가며 답변 품질 비교
- 검색 결과에 출처(어느 문서, 몇 번째 chunk) 함께 표시

**Could**
- 여러 vector DB(Chroma vs FAISS) 간단 비교
- 리랭킹(re-ranking) 개념 맛보기 적용

**트러블슈팅**
- chunk가 너무 크거나 작아서 관련 없는 내용이 검색되는 문제
- embedding 차원 불일치 에러
- 검색은 됐는데 LLM이 여전히 문서를 무시하고 답하는 문제(프롬프트 결합 방식 이슈)

**왜 이 업무를 하는가**
RAG는 현업 LLM 서비스에서 가장 널리 쓰이는 패턴입니다. "회사 고유 데이터를 모델에 안전하게 연결하는 법"을 이해하는 것이 AI 엔지니어의 핵심 업무 중 하나입니다.

**배우게 되는 핵심 역량**
데이터 전처리(chunking), 벡터 검색 파이프라인 설계, 검색-생성 결합 구조 설계

**오늘 배운 AI 개념**
Embedding, Vector Database, 유사도 검색(cosine similarity), RAG(Retrieval-Augmented Generation) 개념

**오늘의 회고**
- 파이프라인(문서→벡터→검색→생성)을 설계하는 과정이 흥미로웠는가?
- 데이터를 다루는 작업과 순수 API 연동 작업 중 어느 쪽이 더 재밌었는가?

## Day 4. RAG 품질 개선 & 비용/성능 최적화 — AI Engineer (Optimization) 역할

**오늘의 회사 및 서비스**
RAG는 동작하지만 대표가 "답변이 느리다", "토큰 비용이 예상보다 많이 나온다", "가끔 엉뚱한 문서를 참조한다"고 지적했습니다.

**오늘의 직무**: AI Engineer (성능/비용 최적화 담당)

**오늘 해결해야 하는 업무**
기존 RAG 파이프라인을 유지보수하며 응답 속도, 정확도, 비용 세 가지 지표를 개선하기.

**실습 목표**
기존 코드를 리팩터링하며 검색 품질 평가, 캐싱, 비용 절감 전략을 실무 관점에서 적용한다.

**Must**
- 질문-정답 세트(테스트케이스 10개) 만들어서 RAG 정확도 수동 평가
- top-k 값, chunk 크기 조정하며 정확도 변화 비교
- 동일 질문 캐싱으로 중복 API 호출 방지 (간단한 캐시 레이어)
- 사용된 토큰 수와 예상 비용을 로그로 기록

**Should**
- 저렴한 모델(mini급)과 고급 모델을 상황별로 분기해서 사용하는 라우팅 로직 설계
- 검색 결과가 없을 때(관련 문서 없음) 적절히 처리하는 fallback 로직

**Could**
- 간단한 평가 스크립트로 자동 채점(정답 포함 여부 체크) 구현

**트러블슈팅**
- 캐시 키 설계 실수로 잘못된 캐시 히트 발생
- top-k를 늘렸더니 오히려 답변 품질이 떨어지는 역설적 상황 경험

**왜 이 업무를 하는가**
실무 AI 엔지니어의 업무 중 상당 부분은 "새 기능 추가"가 아니라 "이미 있는 서비스의 비용/속도/품질 개선"입니다. 이 감각은 튜토리얼로는 배우기 어렵습니다.

**배우게 되는 핵심 역량**
성능 프로파일링, 비용 감각, 트레이드오프 판단(속도 vs 정확도 vs 비용)

**오늘 배운 AI 개념**
평가(evaluation) 개념, 캐싱 전략, 모델 라우팅, top-k와 정확도의 관계

**오늘의 회고**
- 숫자(비용, 속도)를 보며 최적화하는 작업이 성취감을 주었는가, 지루했는가?
- 완성된 것을 "더 낫게 다듬는" 유지보수 작업이 나와 잘 맞는가?

## Day 5. AI Agent 제작 — AI Agent Engineer 역할

**오늘의 회사 및 서비스**
HR팀이 "휴가 신청 가능 일수를 계산해달라", "특정 조건에 맞는 정책을 찾아 요약해달라" 같은, 단순 검색을 넘어선 요청을 하기 시작했습니다.

**오늘의 직무**: AI Agent Engineer

**오늘 해결해야 하는 업무**
LLM이 스스로 "어떤 도구를 쓸지" 판단해서 계산, 검색 등을 조합해 답하는 에이전트를 만들기.

**실습 목표**
LangChain 또는 LangGraph로 도구(tool)를 정의하고, LLM이 상황에 따라 도구를 선택해 실행하는 Agent 구조를 구현한다.

**Must**
- 도구 2개 정의: (1) RAG 검색 도구(Day 3~4 파이프라인 재사용) (2) 휴가일수 계산 등 간단한 파이썬 함수 도구
- LangChain/LangGraph로 Agent 구성, 질문에 따라 적절한 도구 자동 선택되게 만들기
- Agent의 사고 과정(어떤 도구를 왜 선택했는지) 로그로 확인

**Should**
- 도구를 3개 이상으로 확장 (예: 날짜 계산, 간단한 사내 FAQ 검색 등)
- Agent가 도구를 잘못 선택했을 때의 처리(가드레일) 추가

**Could**
- 멀티스텝 추론이 필요한 질문(도구 2개를 순차 사용) 테스트

**트러블슈팅**
- Agent가 도구를 무한 반복 호출하는 루프 문제
- 도구 설명(description)이 모호해서 엉뚱한 도구를 선택하는 문제
- LangChain 버전 이슈로 인한 의존성 충돌

**왜 이 업무를 하는가**
"AI Agent"는 요즘 채용 공고에 가장 많이 등장하는 키워드 중 하나입니다. 단순 Q&A를 넘어 "도구를 활용해 작업을 수행하는 AI"를 설계하는 경험은 실무 트렌드와 직결됩니다.

**배우게 되는 핵심 역량**
Tool/Function calling 설계, Agent 워크플로우 설계, 디버깅(Agent의 판단 과정 추적)

**오늘 배운 AI 개념**
AI Agent, Tool Calling(Function Calling), ReAct 패턴(추론-행동 반복), LangChain/LangGraph 기본 구조

**오늘의 회고**
- Agent가 예상과 다르게 행동할 때, 원인을 추적하는 디버깅 과정이 흥미로웠는가?
- "AI의 판단 과정을 설계한다"는 작업이 다른 날들과 비교해 어떻게 다르게 느껴졌는가?

## Day 6. 외부 도구 연동(MCP) & 통합 트러블슈팅 — Platform/Integration Engineer 역할

**오늘의 회사 및 서비스**
컴퍼스봇을 사내 캘린더, 사내 위키 같은 외부 시스템과 연결해달라는 요청이 들어왔습니다. 마침 회사 표준으로 MCP(Model Context Protocol)를 도입하기로 했습니다.

**오늘의 직무**: AI Platform Engineer

**오늘 해결해야 하는 업무**
기존 Agent에 MCP 서버를 연결하거나, 직접 간단한 MCP 서버를 만들어 Agent가 외부 도구를 표준화된 방식으로 사용하게 만들기.

**실습 목표**
MCP의 개념(왜 필요한가: 도구 연동 표준화)을 이해하고, 간단한 MCP 서버를 직접 만들거나 기존 MCP 서버를 Agent에 연결한다.

**Must**
- MCP가 해결하는 문제(N개 서비스 × M개 도구 조합 문제) 직접 도식화하며 이해
- 간단한 MCP 서버 하나 구현 (예: 사내 FAQ 검색을 MCP 툴로 노출) 또는 공개 MCP 서버 하나 연결
- Day 5의 Agent가 MCP를 통해 도구를 호출하도록 연결

**Should**
- 여러 MCP 서버를 동시에 연결해보고 충돌/우선순위 문제 다뤄보기
- MCP 연결 실패 시의 예외처리(서버 다운, 인증 실패 등)

**Could**
- MCP 서버를 Docker 컨테이너로 패키징

**트러블슈팅 (오늘은 트러블슈팅이 메인 미션)**
- 의도적으로 삽입된 버그: 환경변수 누락, 포트 충돌, MCP 프로토콜 버전 불일치
- Agent-MCP 서버 간 통신 실패 로그를 직접 읽고 원인 추적
- 지금까지 만든 전체 파이프라인(Day1~5)이 새 컴포넌트 추가 후 깨지는 상황을 겪고 복구

**왜 이 업무를 하는가**
실무에서는 "새 기능 개발"보다 "기존 시스템에 새 컴포넌트를 통합하며 발생하는 문제 해결"에 더 많은 시간을 씁니다. 최신 표준(MCP)에 대한 이해도 최근 AI 엔지니어링 채용에서 중요해지고 있습니다.

**배우게 되는 핵심 역량**
시스템 통합, 표준 프로토콜 이해, 로그 기반 디버깅, 장애 대응

**오늘 배운 AI 개념**
MCP(Model Context Protocol)의 목적과 구조, 클라이언트-서버 아키텍처 관점에서의 AI 도구 연동

**오늘의 회고**
- 하루 종일 버그를 추적하고 고치는 작업이 스트레스였는가, 재미있었는가? (→ 인프라 주차와 직접 비교 포인트)
- "표준을 이해하고 맞춰 설계하는" 작업이 나와 맞는가?

## Day 7. 배포, 테스트, 최종 정리 — AI Engineer (Deployment/QA) 역할

**오늘의 회사 및 서비스**
컴퍼스봇이 이제 실제 사내 배포를 앞두고 있습니다. 마지막 관문은 안정적인 배포와 최소한의 테스트/모니터링 체계 구축입니다.

**오늘의 직무**: AI Engineer (배포/QA 담당) — 오늘은 1주일 전체를 마무리하는 날

**오늘 해결해야 하는 업무**
지금까지 만든 서비스(챗봇+RAG+Agent+MCP)를 Docker로 패키징하고, 기본적인 테스트와 간단한 모니터링을 붙여 "배포 가능한 상태"로 만들기.

**실습 목표**
서비스를 재현 가능하게 패키징하고, 최소한의 자동화된 검증 체계를 갖춘다.

**Must**
- 전체 서비스를 Dockerfile / docker-compose로 패키징
- 핵심 시나리오(Day 1~6 테스트케이스) 기준으로 통합 테스트 3~5개 작성
- README 작성 (실행 방법, 아키텍처 다이어그램 포함)

**Should**
- 간단한 로깅/모니터링 (요청 수, 평균 응답시간, 에러율 등) 대시보드화
- CI 관점에서 GitHub Actions로 테스트 자동 실행 설정

**Could**
- 간단한 부하 테스트로 동시 요청 처리 한계 확인

**트러블슈팅**
- Docker 빌드 시 의존성 버전 충돌
- 로컬에서는 되는데 컨테이너 안에서는 API 키를 못 읽는 문제
- 통합 테스트에서 실제 서비스와 다르게 동작하는 부분 발견 및 수정

**왜 이 업무를 하는가**
"만드는 것"과 "배포 가능하게 만드는 것"은 다른 역량입니다. 실무 AI 엔지니어는 모델/파이프라인 설계만큼이나 안정적으로 운영 가능한 형태로 마무리하는 능력이 중요합니다.

**배우게 되는 핵심 역량**
컨테이너화, 통합 테스트 설계, 기본 모니터링, 문서화(README/아키텍처 설명)

**오늘 배운 AI 개념**
LLM 서비스의 배포 파이프라인, 통합 테스트 관점에서의 LLM 평가, 기본 모니터링 지표

**오늘의 회고**
- 1주일 전체를 돌아봤을 때, 어떤 날이 가장 몰입되었는가?
- Docker/테스트/배포처럼 "AI스럽지 않은" 작업도 즐겁게 할 수 있었는가?

# 커리큘럼 현실 반영도 평가

**잘 반영된 부분**
- MVP → 품질 개선 → RAG → 최적화 → Agent → 통합 → 배포로 이어지는 흐름은 실제 스타트업의 LLM 서비스 개발 라이프사이클과 거의 동일합니다.
- 트러블슈팅을 매일 의도적으로 포함시킨 점은 "이론 실습"과 "실무 체험"을 가르는 핵심 차별점입니다.
- 비용/속도/정확도의 트레이드오프(Day 4)를 다루는 것은 실제 AI 엔지니어 업무에서 매우 큰 비중을 차지하지만 튜토리얼에서는 거의 다뤄지지 않는 부분이라 의미 있습니다.

**보완하면 더 현실적인 부분**
1. **협업/커뮤니케이션 요소 부족** — 실무에서는 기획자/PM이 애매한 요구사항을 던지고, 엔지니어가 이를 구체화하는 과정이 많습니다. 하루 정도는 "모호한 요구사항 → 스스로 스펙 정의" 과제를 넣으면 더 현실적입니다.
2. **코드 리뷰 경험 부재** — 실무에서는 동료의 PR을 리뷰하거나 리뷰받는 경험이 많습니다. Claude가 "동료 시니어"로서 코드 리뷰 코멘트를 남기는 형식을 하루에 하나씩 넣어도 좋습니다. (이미 지침에 있는 "코드 리뷰 및 개선" 단계를 매일 명시적으로 강화하면 좋습니다.)
3. **장애 대응(온콜) 시뮬레이션 부재** — 실제 서비스 운영 중 "밤중에 API 비용이 폭증했다", "특정 사용자 질문에서 서비스가 다운됐다" 같은 인시던트 대응 시나리오를 Day 6~7에 추가하면 인프라 주차와의 비교점이 더 선명해집니다.
4. **보안/개인정보 관점 부재** — 1주차가 보안 직무였던 만큼, "사용자가 민감정보를 입력했을 때 로그에 남기지 않기", "프롬프트 인젝션 방어" 같은 미니 과제를 하루에 살짝 넣으면 1주차 경험과 자연스럽게 연결됩니다.
5. **평가(Evaluation) 체계 심화 여지** — Day 4에서 수동 평가만 다루는데, 시간이 되면 LLM-as-a-judge 같은 자동 평가 개념을 Could 과제로 살짝 맛보게 하면 최근 실무 트렌드를 더 반영할 수 있습니다.

원하시면 위 5가지 중 원하는 항목을 골라서 특정 Day에 삽입한 수정본을 만들어드릴 수 있습니다.
